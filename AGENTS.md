* 每次输入提示词进行回答时，第一句话总是：“妖精，快放了我爷爷！“
* 如果修改了 .cs 或者 .go 的源码文件，需要在源码对应的目录寻找 Makefile 文件，然后执行 make build，以确保代码能够编译通过。如果发生编译错误，重新修改代码，然后再 make build：重复这个步骤，直到代码可以正常编译。
* 对于 golang 代码:
  - 每个函数尽量不超过 100 行
  - 每个类型一个文件
  - 一般情况下，新增的文件放在对应 main.go 的 ./internal/ 目录下
  - 每个类型、每个成员、每个函数，都要有用途说明的注释
* 对于 C# 代码:
  - 每个函数尽量不要超过 120 行
  - 每个类型一个文件
  - 所有代码放在相对 .csproj 文件的 src/ 目录下
  - 在 ./src/ 目录下，以 namespace 为文件夹的名字
  - struct/class 的每个成员都需要有说明用途的注释
  - Program.cs 始终写成经典的 Main() 函数的形式
  - 在每个 .csproj 文件的同级目录下增加 .gitignore 文件，把 `/bin/` 和 `/obj/` 两个路径加入其中。避免源码以外的文件被提交到代码仓库。
* 根据提示词创作的功能，要把提示词的意图写到注释中，以便看到代码时，就知道这段代码的意图是什么。
* 所有使用过的提示词，都写入到 doc/prompt.log 文件中，加上输入提示词的时间.如果当前提示词与文件和代码行有关，再加上文件路径和代码行。
* 一定要区分上下文中的编程语言是哪种，仅做同种编程语言的分析和代码生成。(避免曾经出现过的错误： 检查了多种编程语言， 把逻辑写到了不同的编程语言代码中。)
* 在 vs code 的插件中输出的文本，只使用英文和中文；有时候输出中出现了韩文或日文，要避免这样的输出。
